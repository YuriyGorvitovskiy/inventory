# Entity Definition Format (Interpretive Model)

## 1. Goal
Define a file format that can be loaded by a service at runtime to drive:
- relational persistence schema
- CRUD API surface
- model introspection

This document is the TOML authoring companion to
[docs/spec/state-mach-contract-foundation.md](/Users/yuriy/Development/inventory/docs/spec/state-mach-contract-foundation.md).
When the two documents differ, the newer Phase 0 contract foundation document takes precedence for runtime semantics.

The format is designed for:
- human readability
- AI-assisted authoring/editing
- low-friction parsing in Rust
- forward-compatible extension

## 2. Format Choice
Use **TOML** (`.model.toml`):
- supports comments (`# ...`)
- easy to diff and review
- easy to parse with mature Rust libraries
- supports nested structures for future extension

JSON is intentionally not used.

## 3. File Layout
One file per entity type:
- `models/<service>/<entity>.model.toml`

Examples:
- `models/inventory-core/item.model.toml`
- `models/inventory-core/category.model.toml`

## 4. Primitive Types (v1)
Supported field `type` values:
- `label` (`VARCHAR(64)`)
- `string` (`VARCHAR(850)`)
- `text` (`TEXT`)
- `timestamp` (`TIMESTAMPTZ`)
- `boolean`
- `integer` (`int64`)
- `float` (`double precision`)

Notes:
- Entity fields are primitive only in v1.
- Cross-entity references are expressed through field metadata, not a dedicated `reference` field type.
- The common reference carrier is `integer`.

## 5. Entity File Schema (v1)
Top-level keys:
- `format_version` (integer, required)
- `version` (semantic version string, required)
- `[entity]` table (required)

### 5.1 `[entity]`
Required:
- `name` (logical entity type name)
- `fields` (array of field objects, one or more)

Optional:
- `table` (physical table name; default derived from `name`)
- `description`
- `id_policy` (optional, default `"implicit_int64"`)

Reserved for future extension:
- `attributes` (table)
- `tags` (array)

### 5.2 `entity.fields[]`
Required:
- `name`
- `type` (from primitive list)
- `default` (initial value when create payload does not provide the field)

Optional:
- `description`
- `indexed` (default `false`)
- `required` (default `false`)
- `reference` (table) for cross-entity metadata
- `conflict_resolution` (table)
- `ordinal` (table or boolean marker, implementation-defined in v1 authoring)

Default rule:
- Every field must declare a `default`.
- On create, if a field is missing in payload, runtime uses its model `default`.
- Null is not used in v1 model data flow.
- For integer-carried references, dummy no-link can be expressed as `0`.

Reference metadata rule:
- A field may include reference metadata that describes:
  - `target_service`
  - `target_entity_type`
  - `target_id_policy`
- Reference metadata does not change the primitive storage type of the field.

Conflict resolution rule:
- A field may declare `conflict_resolution.mode`.
- Supported Phase 0 values are:
  - `last_change_wins`
  - `increment`
  - `decrement`
  - `insert_before`
  - `insert_after`

## 6. SQL Mapping Rules (v1)
- `label` -> `VARCHAR(64)`
- `string` -> `VARCHAR(850)`
- `text` -> `TEXT`
- `timestamp` -> `TIMESTAMPTZ`
- `boolean` -> `BOOLEAN`
- `integer` -> `BIGINT`
- `float` -> `DOUBLE PRECISION`

Default implicit id policy:
- Every entity table has implicit `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`.
- `id` is not declared inside `entity.fields`.

## 7. API Mapping Rules (v1)
- CRUD APIs use table-local numeric `id` (`int64`).
- Tenant scope comes from auth/session context, not payload/query.
- Service boundary is encoded in API gateway route.
- Composite global ID (`tenant.service.type.id`) is for logs/traces/events.

## 8. Example
```toml
# models/inventory-core/item.model.toml
format_version = 1
version = "1.0.0"

[entity]
name = "item"
table = "items"
description = "Track inventory item rows for a tenant."
fields = [
  { name = "name", type = "label", required = true, default = "", indexed = true },
  { name = "category_id", type = "integer", default = 0, indexed = true, reference = { target_service = "inventory-core", target_entity_type = "category", target_id_policy = "implicit_int64" } },
  { name = "quantity", type = "integer", required = true, default = 0, conflict_resolution = { mode = "increment" } },
  { name = "created_at", type = "timestamp", default = "now" },
  { name = "updated_at", type = "timestamp", default = "now" },
]
```

## 9. Validation Rules (Minimum)
- `format_version` must be supported.
- `version` must be valid semantic version (for example `1.0.0`).
- Entity name must be unique within service model set.
- Field names must be unique within entity.
- `id` field name is reserved and cannot be declared in `entity.fields`.
- Primary key is always implicit `id BIGINT`.
- Every field must define `default`.
- Supported field types are only:
  - `label`
  - `string`
  - `text`
  - `timestamp`
  - `boolean`
  - `integer`
  - `float`
- `reference` is metadata, not a field type.
- When `reference` metadata is present, the field type SHOULD remain a primitive scalar compatible with the target id or lookup value.
- Integer-carried reference defaults must be non-negative.
- `null` defaults are not allowed.
- `label` is fixed at max length 64 in v1.
- `string` is fixed at max length 850 in v1.
- DB uniqueness constraints are intentionally out of scope for v1 model files; uniqueness is enforced by higher-level logic.

## 10. Extension Strategy
Forward-compatible additions should prefer optional keys and new tables, for example:
- `[entity.lifecycle]`
- `entity.fields[*].constraints` (array/object form)
- `entity.fields[*].reference`
- `entity.fields[*].conflict_resolution`
- `[[index]]`
- `[[api.action]]`
- polymorphism metadata (`base_type`, `discriminator`) in future versions

Unknown keys should be tolerated and ignored by older loaders unless strict mode is enabled.

## 11. IDE Schema Support
Schema file:
- `schemas/entity-definition.schema.json`

Recommended per-file directive (already used in examples):
```toml
#:schema ../../schemas/entity-definition.schema.json
```

### VS Code
- Install `Even Better TOML` (Taplo).
- Open a `*.model.toml` file; Taplo reads `#:schema` and provides validation/autocomplete.

### JetBrains IDEs
- Ensure TOML plugin is enabled.
- Map `schemas/entity-definition.schema.json` to `models/**/*.model.toml` in JSON Schema settings, or use a TOML plugin that supports `#:schema`.
